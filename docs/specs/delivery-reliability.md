# 宿主投递可靠性基线

> P5-001.2.3 — Host thin, Vega thick：宿主只负责把 envelope 可靠交到 Vega，不负责 memory intelligence。

## 1. 去重规则

- 宿主在本地缓存和重试队列中，以 `event_id` 为唯一键。
- 同一 `event_id` 在单次会话生命周期内只投递一次。
- Vega 侧仍保留独立去重（canonical dedup），但宿主侧的 `event_id` 去重是可靠性边界的第一道防线。

## 2. 重试参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 初始间隔 | 1 s | 首次失败后等待时间 |
| 最大间隔 | 60 s | 指数退避上限 |
| 最大重试次数 | 3 次 | 含首次投递，共 4 次尝试 |
| 退避策略 | 指数退避（exponential backoff） | 1 s → 2 s → 4 s → ... 封顶 60 s |

- 超过最大重试次数仍未成功的事件，进入离线缓存队列，待网络恢复后继续投递。

## 3. 离线缓存上限

- 离线缓存队列最大容量：**1000 条 envelope**。
- 超出上限时，按 FIFO 丢弃最旧条目，并记录一条 `delivery.dropped` 内部事件。
- 缓存条目至少保留：
  - `event_id`
  - `timestamp`
  - 完整 envelope payload

## 4. 重发顺序保证

- 离线缓存恢复投递时，必须按原始 `timestamp` 升序（先进先出）发送。
- 同一 `session_id` 内的事件顺序必须保持。
- 不保证跨 `session_id` 的全局顺序，但建议宿主在恢复投递时按 session 批次顺序发送。

## 5. 失败判定

- 投递失败 = 网络不可达、HTTP 非 2xx、或超时未收到响应。
- HTTP 4xx（如 400 Bad Request）视为**不可重试错误**，立即丢弃并记录 `delivery.rejected`。
- HTTP 5xx 和超时按上述重试策略处理。

## 6. 一句话版本

宿主保证：同一个 `event_id` 不重复投递，失败时按 1 s / 60 s / 3 次退避重试，离线最多缓存 1000 条，恢复时按原始顺序发送。
