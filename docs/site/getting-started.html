Fast path to a shared long-term memory runtime for coding agents

Getting Started

Install Vega Memory, point it at SQLite and Ollama, store a first memory, and wire the same backend into Cursor or Claude Code. This page uses the current repo scripts, config defaults, and Docker definitions so your next coding agent starts with durable context instead of starting from zero.

Prerequisites

Node.js 20 LTS, a writable local filesystem, and optional Ollama at http://localhost:11434.

Best Fit

Solo developers, small teams, and tooling builders who want multiple coding agents to share the same memory across sessions.

Boundaries

Use Vega as agent memory infrastructure, not as a generic note app, human-first wiki, or project tracker.

1. Installation

Install dependencies, build the project, and expose the CLI locally.

git clone https://github.com/john-hoe/vega-memory.git
cd vega-memory
npm install
npm run build
npm link

2. Configuration

The repo ships .env.example for the core local settings. Add API-specific values when you want HTTP access or the scheduler.

Minimal Local Config

cp .env.example .env
VEGA_DB_PATH=./data/memory.db
VEGA_DB_ENCRYPTION=false
OLLAMA_BASE_URL=http://localhost:11434
OLLAMA_MODEL=bge-m3

Enable HTTP API

VEGA_API_KEY=$(openssl rand -hex 16)
VEGA_API_PORT=3271
export VEGA_API_KEY VEGA_API_PORT
node dist/scheduler/index.js

3. Prove The Smallest Memory Loop

Use the same four-step path the product promises: health, store, recall, then session preload.

CLI

vega health

vega store "Always checkpoint WAL before backup" \
  --type pitfall \
  --project my-project

vega recall "sqlite backup" \
  --project my-project \
  --json

vega session-start --dir "$(pwd)" \
  --mode L1 \
  --json

HTTP API

curl -X POST http://127.0.0.1:3271/api/store \
  -H "Authorization: Bearer $VEGA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "content": "Always checkpoint WAL before backup",
    "type": "pitfall",
    "project": "my-project"
  }'

4. Agent Setup

Pick the first agent surface you want to wire into the same memory backend, then use vega setup --show to confirm the local status.

Codex

Install a managed Vega Memory rules section into ~/.codex/AGENTS.md.

vega setup --codex
vega setup --show

Use this when Codex is your first CLI surface and you want session-start / recall / store guidance installed quickly.

Claude Code

Install a managed Vega Memory rules section into ~/.claude/CLAUDE.md.

vega setup --claude
vega setup --show

Use this when Claude Code is the first client you want to connect through the CLI-first workflow.

Cursor Client Mode

Use the setup helper when Vega is already running as a shared server and you want Cursor to connect in remote client mode.

vega setup --server 127.0.0.1 \
  --port 3271 \
  --cursor

vega setup --show

The manual local MCP stdio example still exists in the deployment guide when you want Cursor to launch the local runtime directly.

5. Docker Deployment

The repository includes a multi-stage Dockerfile and a docker-compose.yml that starts Vega plus Ollama.

Compose

export VEGA_API_KEY=dev-api-key
docker compose up --build -d

curl -H "Authorization: Bearer $VEGA_API_KEY" \
  http://127.0.0.1:3000/api/health

The compose file binds Vega on port 3000 and mounts ./data into the container.

Runtime Notes

services:
  vega:
    command: ["node", "dist/scheduler/index.js"]
    ports:
      - "3000:3000"
  ollama:
    ports:
      - "11434:11434"

For production, replace the default API key, persist the volume, and front the service with your preferred network policy.

6. Product Boundaries

This guide assumes you are adopting Vega as shared memory infrastructure for coding agents.

What Vega Is

One local-first memory backend for agent decisions, pitfalls, preferences, task state, and project context across MCP, CLI, and HTTP workflows.

What Vega Is Not

Not a generic note app, broad knowledge suite, or chat-history replacement where human authoring is the center of the product.
