Choose the runtime shape that matches your adoption stage

Deployment turns the first memory loop into a durable service.

This page exists for the moment after Getting Started works and you need to decide whether Vega should stay local, move into Docker, or become a shared backend for remote machines and multiple coding agents.

Best For

Teams choosing between local-first evaluation, Docker rollout, and shared service deployment.

Decision Inputs

Where the database lives, who needs access, whether remote clients matter, and how much operational control you need.

Next Step

After deployment is chosen, use the API docs to automate client integrations or background workflows.

1. Local Single-Machine

Best when one developer wants the fastest path from setup to repeated session reuse.

Runtime Shape

npm install
npm run build
export VEGA_API_KEY="$(openssl rand -hex 16)"
node dist/scheduler/index.js

SQLite stays local, the scheduler serves HTTP and dashboard, and MCP clients can point at the same machine.

When To Choose It

Choose this when you are still validating product fit, want minimal moving parts, and only need one machine to hold the shared memory runtime.

2. Docker / Compose

Best when you want a reproducible stack with Vega plus Ollama in one place.

Compose

docker compose up --build

curl -H "Authorization: Bearer $VEGA_API_KEY" \
  http://127.0.0.1:3000/api/health

The checked-in Compose stack binds Vega on port 3000 and mounts the local ./data directory.

When To Choose It

Choose this when you want a fast team demo, a reproducible sandbox, or a low-friction bridge between local evaluation and a shared service.

3. Shared Remote Service

Best when multiple developers, machines, or background jobs need the same memory backend.

Cursor Client Mode

vega setup --server 127.0.0.1 \
  --port 3271 \
  --cursor

vega setup --show
vega doctor

Use this when Vega is already reachable as a shared service and Cursor should connect as a remote client instead of launching its own local runtime.

When To Choose It

Choose this when remote machines, team workflows, or scheduled jobs need one authoritative memory store instead of each tool maintaining a silo.

4. Agent Surfaces

Once the runtime exists, connect the surface that matches your workflow today.

Codex

vega setup --codex
vega setup --show

Installs a managed Vega Memory rules section into ~/.codex/AGENTS.md.

Claude Code

vega setup --claude
vega setup --show

Installs a managed Vega Memory rules section into ~/.claude/CLAUDE.md.

5. Deployment Checks

Use the onboarding diagnostics to verify that the chosen runtime shape still matches the agent surfaces around it.

Health And Setup

vega health
vega setup --show
vega doctor

These commands validate runtime health, surface configuration, and the most common onboarding failures before users hit them in real work.
